## Screenshot Specifications

### File Format & Size

- **Format:** PNG (for quality) or WebP (for web optimization)
- **Resolution:** 1920×1080 minimum (full-width captures preferred)
- **File size:** <500 KB (compress without quality loss)
- **Color profile:** sRGB for screen consistency

### Naming Convention

Use this pattern for all screenshot files:

```
[section]_[feature]_[variant].[format]

Examples:
- 5a_theory_card_hover.png
- 5b_search_results_filtered.png
- 5c_detail_page_full.png
- troubleshooting_blank_page_error.png
```

### Storage Location


## Workflow: Creating New Screenshots

### Step 1: Capture the Screenshot

**Tools:**
- Mac: CMD + Shift + 4 (select area), CMD + Shift + 5 (record)
- Windows: Win + Shift + S (Snip tool), or use tool like Snagit
- Online tool: Loom, Gyroflow Toolbox (if video annotation needed)

**Best practices:**
- Capture at native browser resolution (1920×1080 or wider)
- Do NOT include browser chrome (address bar, tabs) unless showing navigation feature
- DO include the app header/navigation
- Use a clean, empty account/state (no personal data visible)
- Highlight UI elements with arrows or boxes if explaining a feature

### Step 2: Prepare the Image

**Crop:**
- Remove unnecessary whitespace
- Crop to relevant UI area
- Include context (surrounding elements help users understand placement)

**Annotate (if needed):**
- Use arrows to point to key UI elements
- Add numbered circles for step-by-step guides
- Use highlight boxes to emphasize areas
- Keep annotations minimal and professional

**Resize:**
- Optimize for web (compress without quality loss)
- Use tools: TinyPNG, ImageOptim (Mac), or online compressors

**Export:**
- PNG for best quality (lossless)
- WebP for smaller file size (more modern browsers)
- Aim for <500 KB per image

### Step 3: Store & Document

**File naming:**
```
[section]_[feature]_[variant].png
Example: 5c_detail_page_full_scroll.png
```

**Storage:**
- Place in appropriate subdirectory under `/docs/assets/screenshots/`
- Commit to git with message: `docs: add screenshot for [feature]`

**Documentation:**
- Note file path in applicable user guide section
- Replace `[INSERT SCREENSHOT: ...]` placeholder with actual filename
- Add alt text for accessibility: `![Description of screenshot showing X feature](...)`

### Step 4: Verify & Deploy

**Checklist:**
- [ ] File meets size requirements (<500 KB)
- [ ] Naming follows convention
- [ ] Stored in correct directory
- [ ] Placeholder in guide updated
- [ ] Alt text is descriptive
- [ ] Committed to git
- [ ] Linked correctly in markdown (use relative paths: `./assets/screenshots/...`)

## Version Control & Updates

### Screenshot Lifecycle

1. **Captured** — Fresh screenshot taken ✅
2. **Optimized** — Cropped, annotated, compressed 🔄
3. **Stored** — Committed to `/docs/assets/screenshots/` 📦
4. **Referenced** — Linked in user guide with alt text 🔗
5. **Current** — Used in published guide ✅
6. **Outdated** — Feature changed, screenshot needs update ⚠️
7. **Archived** — Replaced, but kept for reference history 📜

### Update Trigger

Screenshots should be updated when:
- UI significantly changes (layout, colors, text)
- Feature is redesigned
- New features added
- Platform reaches new major version
- User feedback indicates screenshot is confusing

### Update Process

1. Capture new screenshot with updated UI
2. Name with version suffix if needed: `5c_detail_page_full_scroll_v1.2.png`
3. Replace old file OR keep both (note in comment which is current)
4. Update guide reference to point to new file
5. Commit with message: `docs: update screenshot for [feature] (v1.2)`
6. Optional: Archive old screenshot in `assets/screenshots/_archive/`

## FAQ

**Q: What if I don't have design/graphics skills?**  
A: Use built-in screenshots with basic annotations (arrows, circles). Tools like Loom or Snagit make it easy to add annotations without design skills.

---

**Q: How long should I keep old screenshots?**  
A: Archive for 1–2 major versions, then delete if not needed for historical reference. Keep recent one for comparison.

---

**Q: Can I use video instead of screenshots?**  
A: Videos are great for workflow tutorials! Store in `/docs/assets/videos/`. For a guide, screenshots are usually faster to load and easier to reference.

---

**Q: Should I include usernames/emails in screenshots?**  
A: No. Use placeholder text, demo accounts clearly marked, or blur sensitive data. Never show real user information.

---

**Q: What if the UI keeps changing?**  
A: Prioritize screenshot updates for frequently-changing areas. For stable UI, update quarterly. Use annotations to point out key UI elements (these are less affected by small layout changes).
